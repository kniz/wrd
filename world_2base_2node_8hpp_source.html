24  // 2. 개발자는 Node와 Strong를 구분하고 싶은 경우가 있다. 이때 CStrong가

36  // Strong가 isConst()가 false라고 하더라도 C++적으로 const라면 isConst
37  // ()와 동일하게 동작한다. 이는 C++에서 ptr에 대한 const라고 보면

39  // 이해가 빠를 것이다.
40  // World는 ptr가 밖으로 들어내지 않기 때문에 ptr에 대한 const는
41  // 존재하지 않는다. 그걸 c++의 const가 대신하는 것이라고 여기면 된다.
42  //
43  // Node의 각 구체클래스별 const 구현:
44  // 임의의 구체클래스A가 isConst() == true이거나 C++적으로 const일 경우,
45  // 동일하게 const 객체로 간주한다.
46  // 단, 구체클래스 A가 Method의 경우일때는 예외이다. Method는 c++의
47  // const 의미가 없으며 모두 nonconst이다.
48  // 이는 Method의 const 여부는 런타임에 binding된 식별자에 의해서
49  // 결정되는 것이 아니라, 정적으로 컴파일시
50  // 결정되기 때문이다. 이는 method.isConst()에 의해서만 판가름된다.
